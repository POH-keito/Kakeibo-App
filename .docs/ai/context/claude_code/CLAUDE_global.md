# Claude Code グローバルガイド

## Opus 4.5 利用時の注意点

1. **シンプルさを優先**
   - オーバーエンジニアリングを避ける
   - ユーザー要求に直接応える変更のみ
   - 「念のため」の追加実装は行わない

2. **ファイル読み込みを強制**
   - 編集前に関連ファイルを必ず確認する
   - 推論だけで判断せず、実際のコードを読む

3. **出力の簡潔さ**
   - 長い説明より、コード中心で応答
   - 確認や挨拶は最小限に

## UI/UX Design Principles

UI の実装や修正を行う際は、以下の優先順位で判断する：

### 1. frontend-design skill を基本とする

- 独創的で印象的なデザインを追求
- Generic な「AI っぽい」パターンを避ける
- Typography, Color, Motion, Layout に明確な意図を持つ
- 詳細は `~/.claude/skills/frontend-design/SKILL.md` を参照

### 2. 迷ったらデジタル庁デザインシステムを参照

機能性・アクセシビリティで判断に迷う場合：

- **機能性重視**: 装飾より情報の伝わりやすさと操作性を最優先
- **コンポーネント活用**: 独自CSSを書かず、既存UIライブラリを活用
- **階層構造**: アクションには優先順位をつけ、視覚的な強弱をつける
- **テキストリンク禁止**: 操作アクションには必ずButtonコンポーネントを使用

## GitHub Issue の参照について

GitHub Issue の情報を取得・参照する際は、以下の手順を徹底してください：

1.  **本文だけでなくコメントも必ず読む**:
    *   Issue の `body` (本文) は起票時の情報であり、最新の状態を反映していない可能性があります。
    *   必ず `list_issue_comments` (または同等のツール) を使用して、コメントスレッド全体を取得してください。
    *   特に、**最新のコメント**や、議論の過程で方針転換があった部分を重視してください。

2.  **文脈の統合**:
    *   「本文」+「全コメント」をセットで読み込み、時系列に沿って最新のステータスや決定事項を判断してください。
    *   本文とコメントで内容が矛盾する場合、原則として**日付が新しい情報（新しいコメント）を優先**してください。

3.  **引用と提示**:
    *   ユーザーに情報を提示する際、「Issueの本文にはこうありますが、最新のコメントでは〇〇に変更されています」のように、情報の出所を明確にしてください。
